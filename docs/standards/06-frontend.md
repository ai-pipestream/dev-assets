# 06 — Frontend

This platform has exactly one user-facing frontend, and it lives in one repo. Stop looking for UI in backend services — they don't have one.

## The rule

**All user-facing UI lives in `pipestream-frontend`.** Backend services do not ship their own frontends.

The one small exception: `module-parser` has a Quinoa-served Vue UI at `/admin` that is an internal inspection tool for the parser module itself. It is not the place to build new UI. Don't copy that pattern into other modules.

## `pipestream-frontend` structure

Location: `core-projects/pipestream-frontend/` in the workspace.

It's a pnpm workspace monorepo with:

- `apps/pipestream-frontend/ui/` — the main Vue 3 SPA
- `packages/protobuf-forms/` — shared JSON Schema + JSONForms renderer bindings generated from protos

Tech stack (from `package.json`):

- **Vue 3** + **TypeScript** + **Vite**
- **Vuetify** for base components
- **Pinia** for state
- **connect-es** (`@connectrpc/connect` + `@connectrpc/connect-web`) for gRPC
- **`@bufbuild/protobuf`** for message types
- **JSONForms** (`@jsonforms/core` + `@jsonforms/vue` + `@jsonforms/vue-vanilla` + `@jsonforms/vue-vuetify`) for form rendering

## Talking to backends: connect-es over HTTP/1.1

The frontend uses Connect-Web — same protobuf contracts as native gRPC, but over HTTP/1.1 so browsers can speak it without an envoy proxy. Client setup is uniform:

```ts
// apps/pipestream-frontend/ui/src/services/account-manager/src/services/accountClient.ts
import { createClient } from "@connectrpc/connect";
import { createConnectTransport } from "@connectrpc/connect-web";
import { AccountService } from "@pipestream/protos/account/v1/account_service_pb";

const transport = createConnectTransport({
    baseUrl: window.location.origin,   // routed via Traefik/web-proxy to the right backend
});

export const accountClient = createClient(AccountService, transport);
```

Rules:

- `baseUrl: window.location.origin`. Never hard-code a host. A reverse proxy in front of the frontend routes `/<service>/<rpc>` to the right backend.
- Use `createClient` from `@connectrpc/connect` — not hand-rolled fetch calls, not axios.
- The `AccountService` symbol is imported from the `@pipestream/protos` package, which is generated by the frontend's own `buf.gen.yaml` (see below).
- Errors come back as `ConnectError` with a gRPC-shaped `code` field. Handle them with `try/catch` around `await client.getAccount(...)`.

### Proto codegen for the frontend

The frontend generates its own TypeScript types from the proto sources in `pipestream-protos`. Config lives in `packages/protobuf-forms/buf.gen.yaml`:

```yaml
version: v2
plugins:
  - remote: buf.build/bufbuild/es
    out: gen/ts
    opt:
      - target=ts
```

Rules:

- The generated TypeScript lives under `packages/protobuf-forms/gen/` and is published as a workspace package.
- Never check generated `.ts` files into source control if they're produced by the build (follow the `.gitignore` pattern already in use).
- The frontend does *not* consume Java `grpc-stubs`. Java and TypeScript each generate their own bindings from the same `.proto` files.

## Forms: JSONForms

All dynamic forms are rendered by JSONForms, not by hand-written Vue templates. The reason is that backend services expose JSON Schemas for their configuration DTOs, and the frontend renders those schemas without needing per-service form code.

### Where JSON Schemas come from

Backend services expose schemas at `/q/*` endpoints. For example, a chunker config endpoint might live at `/q/config/chunker` and return:

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "properties": {
    "chunkSize": { "type": "integer", "minimum": 1 },
    "overlap": { "type": "integer", "minimum": 0 },
    "strategy": { "type": "string", "enum": ["fixed", "semantic"] }
  },
  "required": ["chunkSize", "strategy"]
}
```

These are generated server-side from the Java DTO (via Quarkus's OpenAPI schema machinery) and served over plain HTTP. They are **not** derived from proto descriptors — the source of truth for "what config shape does a module take" is the Java DTO that the module uses for its REST developer-convenience endpoint (see the REST exception rule in [`01-protobufs-apicurio-grpc.md`](01-protobufs-apicurio-grpc.md)).

### Rendering with JSONForms

```vue
<!-- UniversalConfigCard.vue -->
<template>
  <JsonForms
    :schema="filteredSchema"
    :data="formData"
    :renderers="renderers"
    @change="handleFormChange"
  />
</template>

<script setup lang="ts">
import { JsonForms } from "@jsonforms/vue";
import { vuetifyRenderers } from "@jsonforms/vue-vuetify";
import axios from "axios";

const schemaEndpoint = "/q/config/chunker";
const filteredSchema = ref<JsonSchema | null>(null);

onMounted(async () => {
  const { data } = await axios.get(schemaEndpoint);
  filteredSchema.value = data;
});
</script>
```

Rules:

- Use `@jsonforms/vue-vuetify` renderers for consistency with the rest of the SPA
- Never hand-write a config form for a backend. If the backend doesn't expose a schema, add the schema endpoint before adding the form.
- Schemas are fetched at runtime, not bundled. This way a backend deployment can ship new config options without a frontend rebuild.

## What about REST from the frontend?

You'll use two kinds of HTTP calls from the frontend:

1. **connect-es** to `AccountService`, `ChunkerService`, etc. — the protobuf contracts
2. **Plain HTTP** to `/q/*` endpoints for JSON Schemas, health, openapi, etc.

That's it. Don't build a third channel (e.g., a custom JSON REST API next to the gRPC one). If a piece of data is needed in the frontend, it goes on the proto service.

## Quinoa or not

The frontend is **not** served through Quinoa in production. It's a pure pnpm monorepo built by Vite and served as static assets by a reverse proxy. Quinoa is only used in module-parser's internal `/admin` UI — which, again, don't copy.

## Platform-registration-service dev container doc correction

Kristian's dev container setup doc for `platform-registration-service` mentions "Vue.js frontend (`src/main/ui-vue`)" — that's a reference to a non-existent directory. `platform-registration-service` is pure backend gRPC with no UI. The dev container doesn't need Node/pnpm for that service; it only needs them for projects that actually have a frontend (i.e. `pipestream-frontend` and the parser's `/admin` page). When the dev container work starts, update the doc to reflect that.
